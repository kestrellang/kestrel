# Issue #7: Try Operator

**Author:** @jkpdino
**Created:** 2026-01-21T20:26:33Z

## Description

## Description

  Add try expressions to Kestrel that allow fallible operations to propagate errors or provide default values in a concise way.

##  Use Case

  Currently, handling errors from fallible functions requires verbose match expressions or explicit error handling. Try expressions would provide a
  more ergonomic way to work with Result types, similar to Rust's ? operator but with Kestrel's own syntax.

  // Example of how you'd like to use this feature
  module Main

  func readConfig() -> Result<Config, Error> {
      // Current approach - verbose
      let fileResult = readFile("config.json")
      let contents = match fileResult {
          Ok(c) => c,
          Err(e) => return Err(e)
      }

      // With try expression - concise
      let contents = try readFile("config.json")

      // Or with a default value
      let contents = try readFile("config.json") else "default"

      parseJson(contents)
  }

##  Proposed Solution (Optional)

  Two forms of try expressions:

  1. Propagating form: try expr - unwraps Ok value or early-returns the Err
  2. Default form: try expr else default - unwraps Ok value or evaluates to default

  The propagating form would only be valid in functions that return a compatible Result type.

##  Alternatives Considered (Optional)

  - Rust-style ? operator - less readable for newcomers
  - Explicit unwrap_or method calls - more verbose
  - Swift-style guard let - requires separate statement

  Additional Context

  Similar features exist in:
  - Rust: ? operator
  - Swift: try? and try!
  - Zig: try and catch

  This would complement Kestrel's existing error handling and make working with fallible operations more ergonomic.
