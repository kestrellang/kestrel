# Issue #107: MIR-3: non-Copyable `let` consumed in one branch drops at if-merge, not scope exit

**Author:** @jkpdino
**Created:** 2026-06-02T16:12:58Z
**URL:** https://github.com/kestrellang/kestrel/issues/107

## Description

## Summary

In MIR-3, a non-`Copyable` `let` that is **consumed on one branch of an `if`** is dropped at the **if-merge block** instead of at **lexical scope exit**. The equivalent `var` is handled correctly (drop driven by an address + drop flag).

## Details

When a `let`-bound non-`Copyable` value is moved out on only one arm of a conditional, drop elaboration inserts the destroy at the control-flow merge point rather than the end of the binding's lexical scope. This means the value can be dropped too early (or in the wrong place) on the not-taken path. `var` bindings already do the right thing because they live at a stack address with a per-binding drop flag; `let` bindings take the SSA path and miss this.

## Repro

`lib/kestrel-test-suite/testdata/memory_model/deinit/diamond_control_flow_move.ks` exercises this (diamond control flow with a conditional move of a non-`Copyable` `let`). The test is currently **skipped** pending the fix.

## Fix direction

Fold into the non-`Copyable`-`let`-via-address work (the consume-half of `let` reads that currently emit copies): give a consumed `let` the same address + entry-block drop-flag treatment that `var` uses, so the drop fires at lexical scope exit guarded by the flag, regardless of which branch consumed it.
