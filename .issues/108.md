# Issue #108: Deque.popFront/popBack leak the extracted element (borrowed-closure return clones, slot abandoned)

**Author:** @jkpdino
**Created:** 2026-06-03T23:36:46Z
**URL:** https://github.com/kestrellang/kestrel/issues/108

## Description

## Summary

`Deque.popFront()` and `Deque.popBack()` leak the element they remove and return. Each pop of a non-`Copyable` element type (e.g. `Deque[String]`) leaks one allocation per call.

This is **independent of** `DequeStorage.deinit` dropping its live elements ‚Äî it's in the element *extraction* path, not the storage drop path.

## Root cause

Both methods extract the element inside a `modify` closure, e.g. `popFront`:

```kestrel
let value = self.storage.modify { (mutating s) in
    let v = s.ptr.offset(by: s.head).read();
    s.head = s.head + 1;
    if s.head >= s.cap { s.head = 0; }
    s.len = s.len - 1;
    v
};
.Some(value)
```

`Pointer.read()` is `where T: Copyable` (a bitwise copy), so `v` aliases the bits already in the ring slot. Returning `v` out of the borrowed `modify { (mutating s) in ‚Ä¶ }` closure forces a **clone** of the non-`Copyable` value (the same "return a value out of a borrowed `with`/`modify` closure ‚Üí clone" behavior seen elsewhere, e.g. the old `RcBox.getValue`). That clone bumps the refcount, but the original slot ‚Äî now abandoned (`head` advanced, `len` decremented, so the deinit only drops the *remaining* `len` elements) ‚Äî is never released. Net: one leaked reference per pop.

## Reproduction

```kestrel
import std.collections.(Deque)

func work(n: Int64) -> Int64 {
    var d = Deque[String]();
    var k: Int64 = 0;
    while k < 4 { d.pushBack("elem \(n) idx \(k)"); k = k + 1 };
    let popped = d.popFront();
    var c = 0;
    if let .Some(s) = popped { c = s.byteCount };
    d.count + c
}

func main() {
    var i: Int64 = 0; var acc = 0;
    while i < 1000 { acc = acc + work(i); i = i + 1 };
    println("done acc=\(acc)");
}
```

`leaks --atExit` reports ~2000 leaks (2 per iteration = the two popped elements), rooted at `work ‚Üí DefaultStringInterpolation.build ‚Üí StringBuilder.build ‚Üí CowBox.init ‚Üí RcBox.init[StringStorage]`. Binding the popped value to a named local and using it (as above) does not help ‚Äî the leak is the abandoned ring-slot reference, not the discarded return value.

## Related / scope

`Array.remove(at:)` uses the same `modify { (mutating s) in ‚Ä¶ removed }` shape and very likely leaks identically; `popLast` / `removeRange` and any other collection method that returns an element extracted inside a `modify`/`with` closure should be audited together.

A clean fix probably wants a real move-out-of-storage primitive (take the element, leave the slot uninitialized) instead of `read()` + closure-return, so no clone is created and no slot is abandoned.

## Context

Found while auditing storage `deinit` element-dropping. The companion deinit fixes (`DictionaryStorage`, `DequeStorage`, `ArrayBuilder` now `dropInPlace` their live elements before deallocating) are correct and separate; this issue is the extraction path only.
